01
Identity in CI/CD pipelines
// Service accounts, workload identity, scoped credentials
In progress

Pipelines need identity. Identity needs boundaries.

CI/CD systems run with privileges that often exceed any single human user. A compromised pipeline credential can push code, deploy to production, or access secrets across every environment.

  • Scoped service accounts: One purpose per credential. The container-push account writes only to the container registry — it does not deploy, read secrets, or access production.
  • Workload identity federation: Replaces long-lived service account keys with short-lived tokens issued at run time. Removes the credentials that get leaked.
  • Least privilege: Pipeline permissions reviewed quarterly. Privilege creep is the primary source of CI/CD blast radius.
  • MFA for human access: Engineers logging into the CI platform require MFA, just as they do for any other privileged system.
📝Knowledge Check0 / 4 answered
Question 1
A CI build agent needs to push container images to a registry. Which approach best follows least privilege?
Select one answer
✓ Correct. Each service account should be scoped to a single purpose. A push-only credential to one registry minimizes blast radius if compromised.
Least privilege means one purpose, one scope. Consolidating service accounts increases blast radius — when one is compromised, everything it can do is compromised.
Question 2
What is the primary security advantage of workload identity federation over static service account keys?
Select one answer
✓ Correct. Workload identity issues short-lived tokens at run time, eliminating the static keys that end up in repos, logs, and config files.
The core benefit is eliminating long-lived credentials. Short-lived federated tokens cannot be exfiltrated and reused later — they expire within minutes.
Question 3
A deployment pipeline service account currently has admin access to your entire cloud account. What should you do?
Select one answer
✓ Correct. Over-privileged service accounts are one of the highest-risk findings in any security audit. Scope down to actual needs and review regularly.
Admin-level pipeline credentials are a top compromise vector. A breached pipeline becomes a breach of every system it can touch.
Question 4
Your CI platform supports human user authentication without MFA "for developer convenience." What is the risk?
Select one answer
✓ Correct. CI platforms are now a primary attack surface. A developer account without MFA on a CI system is equivalent to no MFA on production.
CI platforms have access to deploy, modify pipelines, and reach secrets. They require MFA at the same level as production access.

02
Secrets & credentials
// Storage, rotation, and exposure response
Locked

Secrets in code are not secrets.

Every credential in a repo, log, or environment file is one search away from exposure. The question is not whether secrets will be exposed — it is how fast you detect and revoke them when it happens.

  • Never in source code: Use a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager, Doppler). Inject at runtime.
  • Rotation: Time-bound — at minimum every 90 days, immediately on any suspected exposure.
  • Environment separation: Dev, staging, and production each have their own credentials. A leaked dev credential must not grant production access.
  • Audit trails: All access to the secrets store must be logged.
  • Minimum required permissions: Credentials should authorize only the specific actions they were created for.
📝Knowledge Check0 / 3 answered
Question 1
A developer pushed an API key to a private GitHub repository by accident. They deleted the commit and force-pushed. Is this sufficient remediation?
Select one answer
✓ Correct. Treat any exposed credential as compromised — immediately and unconditionally. Git history can be cached by mirrors, CI systems, and automated scanners.
Once exposed, a credential cannot be unexposed. Automated scanners can capture a secret within seconds of a commit. Always revoke and rotate.
Question 2
A developer wants to test a feature that requires database access. They ask to use the production database credential "just for testing." What is the correct response?
Select one answer
✓ Correct. Environment separation is non-negotiable. Production credentials must never appear in development environments.
Production credentials in development environments are a primary compromise vector. Once present, they can leak through dev repos, dev logs, or dev workstations.
Question 3
What is wrong with passing a database password as a plain-text environment variable to a containerized application?
Select one answer
✓ Correct. Plain-text env vars are visible in docker inspect, Kubernetes pod specs, and process listings. Inject secrets at runtime from a vault instead.
Environment variables are not encrypted at rest and are visible through multiple inspection paths. Use a secrets manager that injects values securely at runtime.

03
Non-interactive authentication
// SSH keys, API tokens, machine-to-machine auth
Locked

Non-interactive credentials never get rotated unless you force it.

SSH keys, long-lived API tokens, and machine credentials lack the natural rotation that human passwords get. They sit unused on disk for years until a breach reveals how broadly they were trusted.

  • One credential per purpose: No shared service accounts spanning multiple systems.
  • Time-bound: Rotation enforced on a schedule, not on convenience.
  • Departed-engineer rotation: Any credential a leaving engineer had access to must be rotated, not just disabled.
  • SSH keys: Private keys never in repos. Public keys logged in a central authority. Personal keys not shared across automation.
📝Knowledge Check0 / 3 answered
Question 1
A service account is used for both database backups and application health monitoring. What is the problem?
Select one answer
✓ Correct. Shared service accounts compound blast radius. A compromise of one usage path gives the attacker all the access the account has across all purposes.
Each service account should serve one purpose. Shared accounts make every compromise vector equivalent to a compromise of the broadest permission set.
Question 2
An engineer who managed several critical service account credentials has left the company. When must those credentials be rotated?
Select one answer
✓ Correct. Departed-engineer credential rotation is mandatory and immediate. Disabling personal accounts is not sufficient — the shared credentials they accessed must also rotate.
Disabling someone's individual account does not revoke their knowledge of shared service credentials. All credentials they had access to must rotate immediately.
Question 3
A deployment script uses an SSH key to connect to production servers. The private key is committed to the deployment repository. What is wrong?
Select one answer
✓ Correct. Private keys in repos are a credential leak waiting to happen. Repo access changes over time, mirrors get created, and history persists indefinitely.
Private keys in repositories are always wrong. Repo access expands over time, history cannot be reliably purged, and CI systems may cache the key in unexpected places.

04
Supply chain security
// Dependencies, SBOMs, and trust transitively
Locked

Every dependency is code you ship — and code you trust.

Your application includes thousands of lines of third-party code from hundreds of upstream maintainers. A compromise of any one of them is a compromise of your application.

  • Dependency review: New packages require evaluation — license, maintainer reputation, recent activity, known vulnerabilities.
  • Pinning and lockfiles: Reproducible builds require pinned versions. A "minor version update" can be a malicious patch.
  • SBOMs: Software Bill of Materials enables you to answer "are we affected?" when a vulnerability is disclosed.
  • Continuous scanning: Dependency scanners flag known CVEs as they become public — not at the next release.
📝Knowledge Check0 / 3 answered
Question 1
A developer adds a new open-source package to the project without a review process. What is the risk?
Select one answer
✓ Correct. Every dependency expands your attack surface. Review for security history, maintainer activity, license, and necessity before adoption.
Popular packages are not automatically safe — supply chain attacks specifically target popular packages because their compromise has maximum reach.
Question 2
Your project has a direct dependency that has not received any updates or security patches in 3 years. What should you do?
Select one answer
✓ Correct. An unmaintained dependency means future vulnerabilities will go unpatched. Plan for replacement before a CVE forces an emergency response.
Unmaintained packages are a known liability. The absence of current CVEs is not safety — it is silence until the next disclosure forces an emergency.
Question 3
What is dependency pinning and why does it matter for security?
Select one answer
✓ Correct. Pinned versions plus lockfile integrity checks ensure the exact same code is built every time, and prevent silent injection of malicious updates.
Pinning is the foundation of reproducible builds and supply chain integrity. Without it, a malicious upstream release can be pulled into your build automatically.

05
SAST — static analysis
// Catching vulnerabilities before they ship
Locked

Static analysis catches what your test suite cannot.

SAST tools read source code to find patterns associated with security flaws — injection sinks, unsafe deserialization, hardcoded secrets, dangerous string concatenations. They run before code ships, which is the cheapest possible time to fix.

  • Run on every PR: Findings caught in review never become deployed vulnerabilities.
  • Block on critical findings: Critical-severity issues fail the build. Lower severities create tickets.
  • Triage false positives: SAST tools generate noise. Suppressions require documented justification.
  • Backlog management: Accumulated medium findings often chain into a single critical exploit path.
📝Knowledge Check0 / 3 answered
Question 1
A SAST scan returns a critical finding: a user-controlled string is being passed directly into a SQL query. The developer says "the front end validates that input." Is this an acceptable mitigation?
Select one answer
✓ Correct. Client-side validation is a UX feature, not a security control. The fix is parameterized queries — front-end checks do not protect the database.
Front-end validation is trivial to bypass. Anyone with a curl command can send arbitrary input to your API. SQL injection fixes belong on the server.
Question 2
Your team has accumulated 47 open SAST findings, mostly medium severity, none blocking. What is the actual risk?
Select one answer
✓ Correct. Chained medium findings are the source of most real breaches. A backlog of "manageable" findings is a chained exploit waiting to be assembled.
Severity ratings score individual findings, not their interactions. Real attackers chain medium issues together to reach impact that no single finding predicted.
Question 3
A SAST finding appears to be a false positive — the vulnerability pattern exists but the surrounding code makes it unreachable. What is the right response?
Select one answer
✓ Correct. Suppressions are fine when justified — but the justification must be documented and reviewed periodically in case surrounding code changes.
Undocumented suppressions accumulate and become invisible risks. Every suppression needs a written reason that a future engineer can review.

06
DAST & runtime security
// Behavioral testing and production safeguards
Locked

Code reads clean, runs broken.

Dynamic Application Security Testing (DAST) exercises the running application — sending real requests, looking at real responses, observing real behavior. It finds what static analysis misses: misconfigurations, authentication bypasses, business logic flaws.

  • Run against staging: DAST sends active attack payloads. It can corrupt data, trigger alerts, or take down services — do not run it against production without explicit coordination.
  • Authenticated scans: Most vulnerabilities are post-authentication. Configure DAST with valid credentials to reach them.
  • Information disclosure findings: Stack traces, server banners, and verbose errors give attackers reconnaissance. Treat as medium-to-high risk.
  • Pair with SAST: Neither tool alone catches everything. Use them together.
📝Knowledge Check0 / 3 answered
Question 1
A SAST scan came back clean on a web application. Is it safe to deploy to production?
Select one answer
✓ Correct. SAST and DAST cover different vulnerability classes. A clean SAST scan rules out certain patterns but does not certify the running application.
SAST is one layer in a defense-in-depth approach. Auth flaws, business logic errors, and config issues are invisible to source-code analysis.
Question 2
Why should DAST not be run against a production environment without specific controls?
Select one answer
✓ Correct. DAST sends real attack traffic. Against production, this can corrupt data, fill alerting queues, or cause outages. Run against staging mirrors.
DAST scans send active attack payloads that production cannot distinguish from real attacks. Use staging environments that mirror production.
Question 3
Your application's DAST scan finds that the server is returning detailed stack traces in error responses. What is the severity?
Select one answer
✓ Correct. Information disclosure is a reconnaissance win for attackers. Stack traces reveal versions, paths, and internal structure that focus subsequent attacks.
Verbose error pages are a reconnaissance gift. They reveal framework versions, file paths, and architecture details that turn blind attacks into targeted ones.
