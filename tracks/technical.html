T1
Identity in CI/CD environments
// Who — and what — is authenticating in your pipeline
In progress

Your pipeline has an identity problem most teams don't think about.

In a CI/CD environment, authentication isn't just humans logging into systems. Build agents, deployment services, container registries, testing frameworks, and infrastructure provisioning tools all authenticate to each other continuously — often using long-lived credentials stored in ways that would be unacceptable for human accounts.

The attack surface is significant: any credential used in an automated pipeline becomes a high-value target. A compromised build agent with the right service account can push malicious code to production, exfiltrate source code, or pivot into cloud infrastructure.

The principle of least privilege in pipelines

Every service account, build agent, and deployment identity should have only the permissions required for its specific function — nothing more. A service account that runs unit tests does not need write access to the container registry. A deployment agent does not need read access to the database. Scoping permissions minimizes the blast radius of a compromised identity.

Human vs. machine identity — different rules

Identity TypeAuthentication MethodKey ControlsCommon Failure
Human developersSSO + MFA + short-lived tokensMFA required, session timeouts, access reviewShared accounts, no MFA on CI platform
Build agentsService account + scoped tokenLeast privilege, token rotation, no interactive loginOver-privileged service accounts, static tokens
Deployment servicesWorkload identity / OIDC preferredEphemeral credentials, audit logging, env scopingLong-lived tokens committed to config files
External integrationsAPI keys / OAuth tokensScoped to minimum permissions, rotated regularlyOrganization-wide admin keys used for single integration

Workload identity — the modern approach

Rather than issuing long-lived credentials to build agents, modern platforms support workload identity federation — the pipeline asserts its identity to a cloud provider using a short-lived, cryptographically verifiable token. The cloud provider validates the assertion and issues a temporary credential scoped to the specific job. No static secret is stored, no rotation is required, and every credential expires automatically.

Question 1
A CI build agent needs to push container images to a registry. Which approach best follows the principle of least privilege?
Select one answer
✓ Correct. Least privilege means each identity gets exactly the permissions it needs — nothing inherited from convenience. Dedicated, scoped service accounts limit what an attacker can do with a compromised build agent.
Each pipeline component should have its own identity with the minimum required permissions. Shared or over-privileged accounts mean a single compromised credential can affect all systems that account touches.
Question 4
What is the primary security advantage of workload identity federation over static service account credentials?
Select one answer
✓ Correct. Ephemeral credentials eliminate the entire class of "leaked long-lived token" incidents. If there is no persistent credential to steal, there is nothing to exfiltrate from logs, config files, or secrets stores. The token that existed for a specific job is gone when the job ends.
The key benefit is eliminating the long-lived static credential. Short-lived, automatically-expiring credentials mean any leaked token is nearly worthless by the time it could be exploited.
Question 7
A deployment pipeline service account currently has admin access to your entire cloud environment "because we weren't sure exactly what it needed." What should happen?
Select one answer
✓ Correct. Over-privileged service accounts are a high-severity finding. The correct response is to determine what is actually needed and scope permissions accordingly. "Monitor for misuse" is not a substitute for least-privilege access control.
An over-privileged pipeline service account represents one of the highest-risk configurations in a software delivery environment. A compromised build agent with admin cloud access can provision attacker-controlled infrastructure, exfiltrate data, and cover its tracks. Remediate immediately.
Question 10
Your CI platform supports human user authentication without MFA "for developer convenience." What risk does this introduce?
Select one answer
✓ Correct. CI/CD platforms are extremely high-value targets precisely because they have trusted access to source code, secrets, and deployment infrastructure. A compromised CI user account without MFA protection can potentially compromise every application the platform builds and deploys.
CI/CD platforms should be treated as critical infrastructure requiring MFA for all human authentication. They hold secrets, have deployment access, and can modify what code runs in production. Attackers specifically target CI systems for this reason.

T2
Secrets & credential security
// How credentials leak and how to stop it
Locked

Credentials in code are a breach waiting to happen.

The accidental commit of API keys, database passwords, private certificates, and access tokens to source control repositories is one of the most common and highest-impact security failures in software development. Public repositories are scanned by automated tools within seconds of a commit — but the problem is just as serious in private repositories, which can be compromised, exported, or made public by mistake.

❌ Do not do this
# credentials hardcoded in application config
DATABASE_URL = "postgresql://admin:Sup3rS3cr3t!@prod-db.internal:5432/app"
STRIPE_SECRET_KEY = "sk_live_4xKb9mNpQ..."
AWS_SECRET_ACCESS_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
✓ Correct approach
# references to environment variables — no credentials in code
DATABASE_URL = os.environ.get("DATABASE_URL")
STRIPE_SECRET_KEY = os.environ.get("STRIPE_SECRET_KEY")
AWS_SECRET_ACCESS_KEY = os.environ.get("AWS_SECRET_ACCESS_KEY")

# secrets are injected at runtime by the platform/vault
# they are never committed to source control

Where secrets leak

Source control commits

The most common vector. Config files, .env files, test fixtures, and migration scripts are frequent offenders. Pre-commit scanning catches this before it reaches the repo.

Build logs

Debug output, error messages, and verbose logging during CI builds can print credential values. Build logs are often stored, shared, and broadly accessible.

Container images

Credentials baked into Docker layers — including in build-time arguments — persist in the image and are extractable even if removed in a subsequent layer.

Environment variable sprawl

Credentials passed as plain-text env vars to containers or serverless functions appear in process listings, deployment manifests, and cloud provider dashboards.

Credential lifecycle requirements

  • Unique per system: One credential per service, per environment. Never reuse credentials across development, staging, and production.
  • Rotated regularly: Non-interactive authentication secrets must be rotated at a minimum annually — and immediately if there is any suspicion of compromise or if personnel with access leave the organization.
  • Rotated within 24 hours of suspected compromise.
  • Stored in vaults or encrypted storage with access restricted to authorized systems and individuals. All access to the secrets store must be logged.
  • Minimum required permissions: Credentials should authorize only the specific actions they're created for — a read-only database credential cannot be used to escalate privileges even if compromised.

Question 12
A developer pushed an API key to a private GitHub repository by accident. They deleted the commit and force-pushed. Is this sufficient remediation?
Select one answer
✓ Correct. Treat any exposed credential as compromised — immediately and unconditionally. Git history can be cached by other systems, pull mirrors, CI systems, and forks. Automated scanning tools may have already captured it. Revoke and rotate first; then investigate exposure scope.
The only correct response to a committed credential is immediate revocation and rotation. Git commits are replicated to CI systems, cached by GitHub's own infrastructure, and scanned by automated tools within seconds. "Quickly deleted" is not a security control.
Question 4
A developer wants to test a feature that requires database access. They ask to use the production database credential "just for testing." What's the correct response?
Select one answer
✓ Correct. Environment separation is a core security control. Production credentials must never appear in development environments — they can be committed to dev repos, logged in dev systems, or exposed through less-secure developer workstations. Separate credentials per environment are non-negotiable.
Production credentials in development environments represent a significant risk. Development systems typically have weaker controls, broader access, and more casual credential handling than production. A credential compromised in dev has production-level access. Issue environment-specific credentials.
Question 7
What is wrong with passing a database password as a plain-text environment variable to a containerized application?
Select one answer
✓ Correct. Plain-text environment variables are visible in `docker inspect`, Kubernetes pod specs, deployment dashboards, and process listings on the host. Prefer injecting secrets at runtime from a vault, or using platform-native secrets management that encrypts values at rest and controls access to reads.
Environment variables are not encrypted. They appear in numerous places that aren't typically considered "secret storage" — container inspection commands, orchestration platform dashboards, cloud provider UIs, and system process listings. Use a secrets management solution that encrypts at rest.

T3
Non-interactive authentication security
// Service accounts, API keys, and machine-to-machine trust
Locked

Machine authentication is held to a higher standard, not a lower one.

Non-interactive authentication — where systems authenticate to each other without a human directly involved — represents a large and often under-managed attack surface. Service accounts, API keys, SSH keys, and authentication certificates used in automated processes must follow stricter controls than human accounts, not looser ones.

Core requirements for non-interactive authentication

  • Single-purpose: A service account used for monitoring must not be used for backups, batch jobs, or anything else. If the account is compromised, the blast radius is limited to that single function.
  • No unrestricted admin: Non-interactive accounts must never have unrestricted administrator or super-user privileges. Full stop.
  • Higher complexity than passwords: API keys, SSH keys, and certificates must be of significantly higher complexity than human passwords.
  • Rotated at minimum annually — and within 24 hours if any person with knowledge of the secret leaves the organization or changes job responsibility.
  • Stored in encrypted vaults with access logging. File system storage with file permissions as the only control is not acceptable for production credentials.
  • Network-scoped where technically feasible: Use firewall rules, IP allowlists, and access control lists to restrict which systems can use a given credential. A database service account credential should only be usable from the application servers — not from a developer's laptop.
  • Monitored for anomalous use: Service account activity should be logged and monitored for behavior that deviates from its expected function.

SSH key management

SSH keys used for automated access (deployment scripts, configuration management, CI pipeline connections to servers) are authentication secrets under the same policy as API keys. Key points:

  • Each automation use case gets its own key pair — never reuse keys across purposes.
  • Private keys must never be committed to source control — not even accidentally.
  • Key rotation is required; many organizations rotate deployment keys on every CI run using ephemeral key generation.
  • Unused SSH keys in ~/.ssh/authorized_keys on servers represent standing access — audit and remove regularly.

Question 9
A service account is used for both database backups and application health monitoring. A security review flags this. Why?
Select one answer
✓ Correct. Multi-purpose service accounts expand blast radius. If an attacker exploits a vulnerability in the monitoring path to compromise the account, they inherit all permissions — including backup access. Single-purpose accounts contain the damage from any single compromise vector.
Shared-purpose service accounts violate least privilege by definition. Compromising the account via any one of its functions grants access to all functions. Single-purpose accounts are a critical blast radius control.
Question 4
An engineer who managed several critical service account credentials has left the organization. When should those credentials be rotated?
Select one answer
✓ Correct. Knowledge of a credential by a departed employee is grounds for immediate rotation — the 24-hour requirement applies. This is distinct from annual rotation schedules. The departing engineer's access window closes when their credentials are rotated, not when their accounts are deprovisioned.
The rotation trigger is change in personnel with credential knowledge — not the annual schedule. Waiting for the next scheduled rotation leaves credentials known to a former employee active for potentially months. 24 hours is the maximum.
Question 7
A deployment script uses an SSH key to connect to production servers. The private key is stored as a file in the CI system's working directory alongside the pipeline code. What's wrong?
Select one answer
✓ Correct. Credentials stored as files in the pipeline workspace are one accidental `git add .` or verbose log statement away from exposure. Platform-native secrets stores encrypt at rest, control access, audit reads, and inject the value as an environment variable or mounted secret at runtime — never as a file in the working directory.
Pipeline workspace files are not secret storage. They can be committed, logged, printed, included in build artifacts, or accessed by any tool running in the same workspace. Use the CI platform's purpose-built secrets management with encryption at rest and access controls.

T4
Software supply chain security
// Dependencies, open source, and the code you didn't write
Locked

Your application is mostly code you didn't write. That's the risk.

Modern applications depend on hundreds or thousands of third-party and open-source packages. Each dependency represents code you're shipping to production that you didn't write, review, or control. Supply chain attacks target this trust relationship — injecting malicious code into widely-used packages or compromising the distribution mechanism itself.

Key supply chain risk vectors

Dependency confusion

Attackers publish packages to public registries with names that match internal private packages. Build systems configured to check public registries first will pull the attacker's version.

Typosquatting

Packages with names that are common typos of popular libraries. One transposed character in an install command pulls malicious code.

Compromised maintainers

Legitimate package maintainers whose accounts are taken over. The attack rides the reputation and existing install base of a trusted package.

Malicious updates

A package that was legitimate when first adopted introduces malicious behavior in a later version. Pinning versions and reviewing changelogs are mitigations.

Developer responsibilities in supply chain security

  • Review before adopting: New third-party packages and open-source components must be reviewed before adoption. Check package provenance, maintainer reputation, download trends, and recent commit history.
  • Pin dependency versions in production builds. Unpinned dependencies allow automatic updates that may introduce vulnerabilities or malicious changes without review.
  • Verify package integrity using checksums or signature verification where supported by the package ecosystem.
  • Scan dependencies for known vulnerabilities as part of the build process — this is a Software Composition Analysis (SCA) scan.
  • Remove unused dependencies. Every package in your dependency tree that isn't actively used is attack surface with no benefit.
  • Treat end-of-life dependencies as high-severity vulnerabilities. An open-source component that is no longer maintained will not receive security patches for new vulnerabilities.
  • Never commit build artifacts, packages, or binaries to source control — use a dedicated artifact repository with integrity verification.

Question 9
A developer adds a new open-source package to the project without a review process "because we need to ship." What risk does this introduce?
Select one answer
✓ Correct. Several high-profile supply chain attacks have used extremely popular packages with millions of weekly downloads. Community adoption is not a security review. Every new dependency requires assessment of known vulnerabilities, provenance, and license before adoption.
Supply chain attacks specifically target popular packages because the blast radius is enormous. Community popularity is not security validation. The review process exists to catch known vulnerabilities, license issues, and suspicious provenance — skipping it for schedule reasons is accepting unquantified risk.
Question 4
Your project has a direct dependency that hasn't received any updates or security patches in 18 months. How should this be treated?
Select one answer
✓ Correct. End-of-life and unmaintained components are categorized as high-severity vulnerabilities by security policy — not because they necessarily have known issues today, but because they will not be fixed when issues emerge tomorrow. Plan migration before the vulnerability occurs, not after.
An unmaintained component is a ticking clock. When a vulnerability is discovered — and it will be — there will be no patch. You're either left vulnerable indefinitely or forced into an emergency migration under pressure. Proactive replacement while the software still works is significantly lower risk and cost.
Question 7
What is dependency pinning and why does it matter for security?
Select one answer
✓ Correct. Unpinned dependencies mean that `npm install` or `pip install` today may pull a different package version than it did yesterday — potentially one with newly introduced malicious code. Pinning ensures every build pulls exactly the same code, making changes reviewable and intentional.
Without pinning, your build process may silently pull updated package versions that contain vulnerabilities or malicious changes. Pinning makes dependency updates a deliberate, reviewable act rather than an automatic, invisible one.

Static Application Security Testing (SAST)
// Finding vulnerabilities before the code runs
Locked

SAST finds what code reviews miss — automatically, at scale, before deployment.

Static Application Security Testing analyzes source code, bytecode, or binaries without executing the application. It identifies classes of vulnerability — injection flaws, insecure cryptographic practices, hardcoded credentials, unsafe API usage — by examining code structure and data flow patterns.

What SAST finds

Vulnerability ClassExampleTypical Severity
Injection (SQL, LDAP, command)User input concatenated directly into a queryCritical
Hardcoded credentialsAPI keys, passwords in source codeCritical
Insecure cryptographyUse of MD5/SHA1 for password hashing, ECB modeHigh
Insecure deserializationDeserializing untrusted data without validationHigh
Path traversalUser-controlled file paths without sanitizationHigh
Insecure randomnessUsing predictable random for session tokens or keysMedium
Missing input validationUnvalidated user input reaching sensitive operationsMedium

SAST in the development lifecycle

  • Shift left: SAST is most valuable when integrated early — in the IDE during development, in pre-commit hooks before code reaches the repository, and in CI pipelines on every pull request. Findings caught in development cost a fraction of those found in production.
  • Build gate integration: Critical and high severity findings should block merges or deployments. Teams that allow critical SAST findings to ship accumulate technical security debt that becomes increasingly expensive to remediate.
  • Remediate before release: All critical and high severity SAST findings must be addressed before release. Document and justify false positives with technical rationale — do not simply suppress findings without documented review.
  • SAST is not sufficient alone: Static analysis finds structural vulnerabilities in code but cannot detect configuration issues, runtime behavior, or vulnerabilities that emerge from how components interact. It must be combined with other testing approaches.

False positives — a real challenge

SAST tools generate false positives — findings that are flagged as vulnerabilities but are not actually exploitable in context. The correct response is to review and document the justification, not to suppress or ignore the finding. A suppression without documented rationale is a future liability — when something similar turns out to be a real vulnerability, there's no record of why it was dismissed.

Question 9
A SAST scan returns a critical finding: a user-controlled string is being passed directly to a SQL query without parameterization. The developer says the input is "validated on the front end." Is this finding a false positive?
Select one answer
✓ Correct. Front-end validation is a UX control, not a security control. Any attacker can bypass the browser entirely and send arbitrary data directly to the API. SQL injection prevention requires server-side parameterized queries — the input pathway doesn't matter.
Never rely on front-end validation for security. HTTP requests can be crafted by any client and sent directly to your endpoints, completely bypassing browser-side validation. The only valid SQL injection control is server-side parameterized queries or prepared statements.
Question 4
Your team has accumulated 47 open SAST findings, mostly medium severity, none blocked by the build gate. They've been open for 6 months. What risk does this represent?
Select one answer
✓ Correct. Medium findings can chain — an information disclosure finding combined with an authentication weakness can form a critical exploit path. Beyond the technical risk, a backlog of 47 unaddressed findings signals that the security testing process is producing findings no one acts on — which undermines the entire program.
Accumulated security debt compounds. Multiple medium findings can combine into critical exploit chains. More broadly, a large unaddressed SAST backlog signals that findings aren't driving behavior change — which means the scanning process is producing noise, not security improvement. Address findings systematically.
Question 7
A SAST finding appears to be a false positive — the vulnerability pattern exists in the code but the data flow that would make it exploitable doesn't. What is the correct way to handle this?
Select one answer
✓ Correct. False positives require documented, reviewed dispositions — not silent suppression. The documented rationale protects your team when auditors review findings, and ensures that if the pattern later becomes exploitable (due to a code change), there's a record of the prior assessment to review and update.
Suppressing findings without documentation creates invisible risk. Code evolves — a pattern that's not exploitable today may become exploitable when another component changes. Without a documented disposition, future reviewers have no way to know the finding was assessed rather than overlooked.

T6
DAST & runtime security testing
// Finding vulnerabilities in running applications
Locked

Static analysis sees the code. Dynamic testing sees what the code actually does.

Dynamic Application Security Testing (DAST) tests a running application by sending crafted requests and analyzing responses — simulating how an attacker would interact with the application from the outside. Unlike SAST, DAST doesn't require access to source code and is language-agnostic.

SAST vs DAST — complementary, not competing

DimensionSASTDAST
What it testsSource code structure and patternsRunning application behavior
When in lifecycleDevelopment, pre-commit, CIPre-release, staging, production
Source code requiredYesNo
Runtime config issuesCannot detectDetects
Authentication flowsLimited detectionCan test fully
Business logic flawsVery limitedBetter coverage
False positive rateHigherLower (observed behavior)

What DAST uniquely finds

  • Authentication and session management flaws: Session fixation, session tokens that don't expire, predictable token values, authentication bypass through parameter manipulation.
  • Security header misconfigurations: Missing or misconfigured Content-Security-Policy, X-Frame-Options, HSTS, and cookie security attributes.
  • Server-side request forgery (SSRF): Can the application be manipulated into making requests to internal infrastructure?
  • Exposed sensitive endpoints: Admin interfaces, debug endpoints, internal API paths inadvertently accessible from the public-facing application.
  • Error handling and information disclosure: Stack traces, connection strings, and internal paths revealed in error responses.
  • TLS and transport security issues: Weak cipher suites, expired certificates, mixed content.

Running DAST responsibly

DAST sends active attack payloads to a running application. This means:

  • Never run DAST against production without explicit authorization and controls in place to prevent test data from contaminating real data.
  • DAST is typically run against a production-equivalent staging environment.
  • Inform operations teams before running DAST — active scanning can look like an attack and trigger incident response.
  • Scan results should be reviewed for both the reported findings and any unexpected application behavior observed during the test.

Secure development lifecycle integration

A mature security testing program integrates multiple layers:

  • SAST on every commit and PR (shift-left, fast feedback)
  • SCA (Software Composition Analysis) scanning for known vulnerable dependencies on every build
  • Secret scanning to prevent credential commits
  • DAST on every release candidate in staging
  • Penetration testing periodically and for major releases

Each layer catches different classes of vulnerability. Removing any one layer creates a blind spot. The goal is defense in depth — multiple overlapping controls so that a finding missed by one method is likely caught by another.

Question 9
A SAST scan came back clean on a web application. Is it safe to deploy to production?
Select one answer
✓ Correct. SAST is one layer of a defense-in-depth testing strategy. It finds code-level structural issues but cannot detect runtime configuration vulnerabilities, authentication bypass through legitimate flows, security header misconfigurations, or business logic flaws. A complete pre-production process requires DAST, SCA, and potentially penetration testing.
SAST covers code structure but has significant blind spots. Authentication flows, session management, security response headers, server-side request forgery, and misconfigured infrastructure are all invisible to static analysis. A clean SAST scan is necessary but not sufficient.
Question 4
Why should DAST not be run against a production environment without specific controls in place?
Select one answer
✓ Correct. DAST fuzzes the application with malformed, malicious, and boundary-condition inputs. Against a production database, test records may be created or real data corrupted. Active scanning traffic may trigger security incident response. Most DAST testing is performed in a staging environment that mirrors production configuration.
DAST is active — not passive. It deliberately sends attack payloads to the application. In production, this risks data corruption, service impact, and false security alerts. DAST against production requires authorization, data isolation controls, and coordination with operations teams.
Question 7
Your application's DAST scan finds that the server is returning detailed stack traces in 500 error responses, including file paths and internal hostnames. What severity is this and why?
Select one answer
✓ Correct. Information disclosure through error messages is typically medium severity but has significant impact on attacker effectiveness. Internal paths reveal the framework and language, file system structure aids path traversal attempts, and internal hostnames support network reconnaissance. This is exactly the intelligence an attacker uses to tailor subsequent attacks.
Error information disclosure is classified as medium-to-high severity because it functions as reconnaissance intelligence. While it doesn't directly enable exploitation, the framework name, version, file paths, and internal hostnames all help an attacker tailor subsequent attacks to your specific environment. Production error responses should return generic messages with a reference ID — detailed errors belong only in internal logs.
